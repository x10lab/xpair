## CONTRADICTS

- No-client Host completion / no-client Done / no-client Host ready / no-client serving start. Representative 예상: `client 없이 Done 단계로 이동`, `Host ready Done으로 이동하고 나중에 pair 가능하다고 안내`, `Open Xpair 완료 버튼을 누름 → Host onboarding을 완료하고 no-client Host ready 상태로 전환`, `완료 처리가 시작되면 ... Host 준비 전환으로 이어져야 함`. Evidence: §1.5 (Q0543) says, "Starting XpairHost before any client is acceptable, but with no connected client the Host onboarding is expected to hold at the permission step rather than report completion." Conflict: these expectations report/completed-ready a Host flow without a connected client.
- Engine choice/status after host setup, reconnect, password, host reachability, or permission gates. Representative 예상: `Retry 성공 후 사용자가 Next를 누름 → Engine 단계로 이동`, `Reconnect host 성공 상태에서 사용자가 Next를 누름 → Engine, 권한, 파일 접근 온보딩으로 이동`, `AX와 SR이 모두 승인되고 사용자가 Next를 누름 → Engine 단계로 이동함`, `host 연결 확인 후 다음 온보딩 단계로 이동`, where the next step is Engine. Evidence: §1.3 (Q0545) says, "Engine choice (Claude / Codex / OpenCode) should be offered **before** the device-name / host-setup step, so the user picks their agent up front rather than after pairing." Conflict: these expectations put Engine after host setup/pairing prerequisites.

## BACKED

- Xpair has separate Host and Client roles. Representative 예상: `Host 완료와 client 실행은 분리되어야 함`, `Xpair client workbench 흐름으로 이동`, `Host-first 흐름`. Evidence: §0.2 (Q0343) says, "Xpair has two different app roles: Host and Client. They should not be collapsed into a single identity because the Host and Client serve different jobs."
- Host is the permission-holding side. Representative 예상: permission and Host setup branches. Evidence: §0.2 (Q0245, Q0337, Q0443) says, "Host is the permission-holding side. It is expected to run on the machine being controlled and to hold the macOS grants needed for computer-use."
- Client onboarding is pre-workbench and not a workbench tab. Representative 예상: `workbench 복귀 가능 상태로 이동`, `Xpair client workbench 흐름으로 이동` only after onboarding progression. Evidence: §1.2 (Q0369, Q0421, Q0424, Q0426) says, "Client onboarding appears before the IDE workbench. The workbench window should not appear alongside it, and onboarding should not be an editor tab."
- Client onboarding opens the intended surface only after setup. Representative 예상: `온보딩 미완료로 종료`, `완료 없이 종료`, `workbench가 선택한 세션으로 열림`. Evidence: §1.2 (Q0369, Q0402, Q0474) says, "Client onboarding closes only after the necessary setup is complete, then the IDE opens into the intended working surface."
- Host onboarding must exist for permission/TCC setup. Representative 예상: `Host onboarding에서 권한...`, `XpairHost 권한 부여 안내`, permission-gate branches. Evidence: §1.2 (Q0441, Q0442, Q0443) says, "Host onboarding must exist. It is responsible for getting the Host through the required permission/TCC flow."
- Unresolved TCC blocks Host usability. Representative 예상: `권한 미완료라 host setup을 중단함`, `Next 비활성`, `setup을 완료하지 않음`. Evidence: §1.5 (Q0443) says, "Host onboarding must resolve required macOS permissions before the Host is considered usable. If TCC is not resolved, the app should not proceed as though setup succeeded."
- Permission actions can reopen onboarding steps. Representative 예상: permission retry/reopen branches. Evidence: §1.2 (Q0473, Q0493, Q0494) says, "Permissions and Settings actions should be able to reopen the relevant onboarding step instead of opening disconnected UI."
- `xpair` CLI availability is a hard gate. Representative 예상: `CLI 의존 단계가 계속 막힘`, `CLI 준비 전까지 manual 입력은 막힘`, `CLI 미준비 상태로 온보딩 미완료 종료`. Evidence: §1.3 (Q0533, Q0534, Q0536, Q0537) says, "`xpair` CLI availability is a hard product requirement before flows that need it. The onboarding must either install it before the hard gate or block with a clear reason."
- Agent engine checks are required when selected. Representative 예상: `Claude Code가 설치와 인증을 모두 통과함`, `Codex가 설치와 인증을 모두 통과함`, `opencode가 설치와 인증을 모두 통과함`. Evidence: §1.3 (Q0541) says, "If the user chooses Claude, Codex, or OpenCode support, onboarding should check for that tool and help install/configure required environment variables."
- Bonjour LAN discovery first. Representative 예상: `Bonjour LAN host`, `scan 완료 후 발견된 host`, `Discover scan`. Evidence: §1.4 (Q0382, Q0384) says, "First connection should be LAN-first: scan the local network with Bonjour and offer to connect when another Mac is found."
- Tailscale fallback and no-host guidance. Representative 예상: `같은 Wi-Fi, Tailscale, 수동 연결 fallback`, `Tailscale 안내`, `no-host 안내`. Evidence: §1.4 (Q0383, Q0384) says, "Tailscale is a fallback, not a prerequisite. If no same-network Mac is found, the product should naturally guide the user toward Tailscale or another fallback path."
- Host install from client onboarding is in roadmap scope. Representative 예상: `XpairHost 미설치 host로 보고 계정/비밀번호 입력 단계로 이동`, `XpairHost 다운로드와 설치 단계로 이동`. Evidence: §5 M2 (Q0382, Q0383, Q0384, Q0440, Q0515, Q0525) says, "**M2: Install and pairing** - Xpair naming, role-aware install, LAN Bonjour discovery, Tailscale fallback, host install from client onboarding."
- Sessions are the product center and must be attachable. Representative 예상: `host session 생성이 성공함`, `attach가 성공함`, `기존 세션 workbench로 이동`. Evidence: §1.6 (Q0056, Q0153, Q0154) says, "The product is centered on launching/attaching persistent host sessions from the client."
- Folder mapping is a real flow and Browser should expose mapping state/Add Mapping. Representative 예상: `mapping 검증`, `프로젝트 선택`, `Add Mapping`-style branches. Evidence: §1.7 (Q0041, Q0042, Q0043) says, "Xpair assumes files are available on the host and maps client paths to host paths." Evidence: §1.7 (Q0414) also says, "Client UX should use **Add Mapping**, not generic **Open Folder**."
