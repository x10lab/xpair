## BACKED

- LAN/Bonjour discovery exists and should offer a connection path when another Mac is found. This backs expectations such as Bonjour LAN candidates leading to connection/setup choices at a high level, but not the exact `Connect/Reconnect/Set up host` taxonomy. Evidence: §1.4 (Q0382, Q0384) says "First connection should be LAN-first: scan the local network with Bonjour and offer to connect when another Mac is found."
- Tailscale is a fallback path, not a prerequisite, so expectations that guide the user toward Tailscale only when LAN/same-network discovery is insufficient are backed at a high level. Evidence: §1.4 (Q0383, Q0384) says "Tailscale is a fallback, not a prerequisite."
- Discovery must account for tailnet situations, so checking fallback behavior around tailnet/MagicDNS is backed at a high level. Evidence: §1.4 (Q0399) says "The product should verify that discovery actually works on the user's likely topology, including tailnet situations where MagicDNS may be off."
- Host onboarding must resolve macOS permissions before the Host is usable, backing expectations that unresolved AX/SR-style permission steps block success. Evidence: §1.5 (Q0443) says "Host onboarding must resolve required macOS permissions before the Host is considered usable."
- Host onboarding exists as a TCC/permission flow, backing expectations that permission setup is a real Host-side onboarding step. Evidence: §1.2 (Q0441, Q0442, Q0443) says "Host onboarding must exist. It is responsible for getting the Host through the required permission/TCC flow."
- Engine support should include Claude, Codex, and OpenCode checks when selected, backing the high-level expectation that choosing one of those engines triggers a tool/configuration check. Evidence: §1.3 (Q0541) says "If the user chooses Claude, Codex, or OpenCode support, onboarding should check for that tool and help install/configure required environment variables."
- Engine choice must exist before pairing/host setup, backing expectations where `claude`, `codex`, and `opencode` are presented as supported engine choices before Connect. Evidence: §1.3 (Q0545) says "Engine choice (Claude / Codex / OpenCode) should be offered **before** the device-name / host-setup step".
- Client onboarding should not complete until necessary setup is done, backing the high-level outcome that incomplete auth/setup should not be treated as successful onboarding. Evidence: §1.2 (Q0369, Q0402, Q0474) says "Client onboarding closes only after the necessary setup is complete".
- Onboarding should stay outside the IDE workbench, backing expectations that VSCodium/workbench surfaces are not available as alternate paths during onboarding. Evidence: §1.2 (Q0369, Q0421, Q0424, Q0426) says "Client onboarding appears before the IDE workbench. The workbench window should not appear alongside it, and onboarding should not be an editor tab."
- Host key fingerprint should be hidden until expanded, backing expectations where the user explicitly opens host-key details to inspect the fingerprint. Evidence: §1.2 (Q0430) says "Host key fingerprint should be hidden by default and revealed only when expanded."
- Host install from client onboarding is a roadmap-backed direction, backing only the high-level existence of a client-driven host-install path. Evidence: §5 M2 (Q0382, Q0383, Q0384, Q0440, Q0515, Q0525) says "host install from client onboarding."
- The supported host/client path is Apple Silicon Mac, backing the high-level expectation that arbitrary non-Mac SSH targets are outside the supported path. Evidence: §2 says "Target platform is Apple Silicon Mac for the supported host/client path unless the user explicitly expands scope."
